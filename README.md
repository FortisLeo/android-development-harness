# Android Agent Harness

A device-validation harness for OpenCode/Hermes-driven Android development. It uses a real Android device over ADB and collects screenshots, accessibility hierarchy, logcat, device metadata, and test output.

## Current status

OpenCode is installed at `~/.local/bin/opencode`. Android SDK tools are installed at `/opt/data/android-sdk`. Connect a device with USB debugging enabled, accept the RSA prompt, and verify it with:

```bash
source ~/.profile
scripts/android_device.sh status
```

For multiple devices, select one explicitly:

```bash
export ANDROID_SERIAL=<device-serial>
```

## Commands

```bash
scripts/android_device.sh status
scripts/android_device.sh install path/to/app-debug.apk
scripts/android_device.sh launch com.example.app
scripts/android_device.sh collect com.example.app artifacts/manual
scripts/android_device.sh reset com.example.app
```

`collect` writes:

- `screenshot.png`
- `accessibility.xml`
- `logcat.txt`
- `windows.txt`
- `properties.txt`
- `package.txt`
- `meminfo.txt`
- `metadata.json`

## UI automation

The UI automation framework is Maestro. It is used for readable end-to-end flows on a real Android device or emulator. UI Automator accessibility XML is the secondary semantic validation layer, and screenshots are collected at each important flow state.

The lifecycle is deliberately split into two phases:

1. Coding phase — implement the feature, create/update Maestro flows and accessibility assertions, run Gradle/unit/instrumentation/lint checks, and build the debug APK. Do not run Maestro or device UI validation here.
2. Final validation phase — only after coding checks pass, install the APK, run Maestro on a connected device/emulator, collect screenshots/accessibility/logcat, compare baselines, and create a baseline only after a passing flow.

A UI feature is not complete until both phases pass. If final validation fails, fix the root cause, rerun coding checks, then rerun final validation, with a maximum of three cycles.

## Baseline policy

Baselines are created only after a Maestro flow succeeds. A connected device or emulator is mandatory. `android_create_baseline_from_flow` fails if there is no ready device, Maestro is unavailable, or the flow fails. The flow should use `takeScreenshot` after every important screen; the baseline tool records the final screen plus accessibility state, while per-screen screenshots can be generated by Maestro debug output or explicit screenshot steps.

## Files

- `scripts/android_device.sh` — ADB device operations and artifact collection
- `scripts/validate.sh` — install, reset, launch, collect, and crash-marker gate
- `scripts/build_and_validate.sh` — build an Android repository and validate its debug APK
- `scripts/screenshot_diff.py` — dependency-free exact PNG check
- `scripts/common.sh` — macOS/Linux SDK environment and shared safety checks
- `flows/` — optional Maestro flows
- `screenshots/baseline/` — committed expected screenshots
- `artifacts/` — generated, ignored validation output
- `config/device.env.example` — local device/project configuration template
- `mcp_server/android_mcp.py` — local MCP server exposing Android device/testing tools
- `mcp_server/README.md` — MCP setup and tool reference

## OpenCode workflow

From an Android repository, run:

```bash
opencode run --agent build \
  'Implement the requested feature. Build and test it. Use the Android agent harness at ~/Projects/android-agent-harness to install the APK on the connected device, collect accessibility/screenshot/logcat artifacts, inspect failures, and repair them. Do not stop until validation passes or three repair cycles are exhausted.'
```

The harness deliberately does not assume a package name or APK path; pass those from the target repository.
